# 幻灯片生成规格说明文件编写 Skill

本 Skill 指导 Agent 通过**访谈式对话**，与用户共同挖掘并澄清内容，最终整理成一份**幻灯片生成规格说明文件**（Markdown），作为后续幻灯片生成 Agent 的唯一权威输入。

> **核心原则：先访谈，后生成。** 不要拿到主题就直接写规格，而要通过多轮追问，帮助用户理清思路，确认内容后再动笔。

---

## 关键定位

- **当前任务**：通过访谈澄清内容，生成 Markdown 规格说明文件，**不是直接生成 PPTX**。
- **后续 Agent 任务**：根据该规格文件生成正式可编辑幻灯片。
- 规格文件一旦确认，后续 Agent **不得修改**页面标题、副标题、内容、Speaker Notes、顺序、角色。

---

## 访谈驱动的工作流程

### 阶段一：启动访谈（必须先完成，再进入后续阶段）

用户提出主题后，**不要立刻生成规格**。使用 `AskUserQuestion` 工具，按以下顺序分多轮追问，每轮最多问 2–3 个问题，等用户回答后再继续下一轮。

**第一轮：摸清场景与目的**
- 这个 PPT 是给谁看的？（职位、背景、决策权）
- 在哪个场合使用？（内部汇报、客户演讲、大会分享、异步阅读……）
- 你希望听众看完后，做什么或相信什么？（核心目的）

**第二轮：挖掘核心内容**
- 你最想让听众记住的一句话判断是什么？
- 目前有哪些素材、数据或案例可以支撑这个判断？
- 有没有你认为"必须讲"的内容，以及"绝对不能讲"的内容？

**第三轮：厘清叙事逻辑**
- 为什么现在要讲这个话题？（背景变化或触发事件）
- 听众当前面临什么问题或挑战？
- 你准备提出什么解法或建议？

**第四轮（按需）：确认边界与约束**
- 大约几页合适？有没有时间限制？
- 是否需要产品映射页、路线图页、附录？
- 哪些信息还没确认（待核实的数据、客户案例、对外口径）？

> **访谈原则**
> - 不急于给方案，先帮用户把问题想清楚。
> - 用户给出模糊答案时，继续追问，直到核心叙事和关键内容都足够具体。
> - 访谈信息足够后，向用户确认："我已经整理了足够的信息，准备开始写规格文件，你确认吗？"

---

### 阶段二：创建任务列表

访谈完成后，使用 `TaskCreate` 跟踪以下任务：
1. 幻灯片任务说明
2. 核心叙事主线
3. 总体结构规划
4. 逐页规格编写
5. 自检与交付

---

### 阶段三：生成规格文件

在 outputs 目录创建 `.md` 文件（如 `slides-spec-{topic}.md`），严格按以下章节结构填写。

---

## 必须固定的内容

1. 隐藏说明页（正式 deck 第一页，PPTX 中设为隐藏）
2. Thank You / 感谢页（最后一页）
3. 故事主线
4. 页面结构（页数、角色、承接关系）
5. 每页标题、副标题、主张、内容、注意事项
6. 每页 Speaker Notes（可直接照念）
7. 范围边界（已确认 / 待确认 / 不在范围）
8. 后续生成参考与约束

---

## 规格文件结构（按以下章节组织）

### 1. 幻灯片任务说明
- 幻灯片主题
- 目标听众
- 使用场景
- 核心目的
- 输入材料与范围边界（表格：输入材料 / 已确认 / 待确认 / 不在范围）

### 2. 核心叙事主线
用一句话回答："为什么讲？问题为什么重要？为什么相信这个判断？接下来怎么办？"

推荐模板：背景变化 → 矛盾挑战 → 影响后果 → 关键判断 → 解决思路 → 行动建议

合格主线需有：因果、递进、取舍、结论、行动。

### 3. 术语与表达规范
- 推荐术语表（术语 / 含义 / 使用建议）
- 避免使用的表达
- 语言风格：中文为主，专业克制，短句优先，默认微软雅黑

### 4. 幻灯片总体结构
表格列出每页：页码 / 页面标题 / 页面角色 / 逻辑作用 / 是否展示。

默认结构：第 1 页隐藏说明页 → 封面 → 主线页... → 行动建议页 → Thank You。页数服务于故事，**不套用固定页数**。

### 5. 逐页规格说明
每页严格按以下结构编写：

```markdown
## 第 X 页：【页面标题】

### 页面角色
【这一页在故事中承担什么任务】

### 页面主张
> 【一句话说明听众要记住什么】

### 页面标题
【正式标题，确认后不得修改】

### 页面副标题
【可选；确认后不得修改】

### 页面内容
【bullet / 表格 / 流程 / 卡片 / 对比；确认后不得改写、扩写、删减、重排】

### Speaker Notes
\`\`\`text
这一页我们主要想说明……
前面一页讲到……，所以这里进一步看……
页面上有三个重点：第一……第二……第三……
因此结论是……自然引出下一页……
\`\`\`

### 页面草图 / Wireframe
\`\`\`text
【文本框表达信息层级，不追求美观】
\`\`\`

### 与前后页面的关系
- 承接上一页：……
- 推动下一页：……

### 设计意图
【为什么这样组织】

### 注意事项
- 哪些内容不要放入
- 哪些必须保留
- 哪些表达需统一
- 未确认内容标注【待确认】
```

### 6. 后续生成阶段的约束（写入规格文件，提示后续 Agent）

**硬约束（不得违反）**：
1. 严格按规格文件生成，不修改标题/副标题/内容/Speaker Notes/顺序/角色
2. 不自行新增、拆分、合并页面
3. Speaker Notes 写入演讲者备注区
4. 第一页设为隐藏幻灯片，最后一页为 Thank You

**视觉与格式**：
- 16:9 宽屏，浅色背景，微软雅黑（除非另有指定）
- 必须可编辑 .pptx，不得用 PDF 或整页图片
- 表格必须用原生表格，禁止用文本框/形状/线条拼接

**待确认信息**：产品名称、客户案例、外部数据、对外口径如未校准，标注【待确认】，后续 Agent 不得自行替换或编造。

---

## 常见页面类型（按需使用）

- **隐藏说明页**（必有）：创作背景、目标听众、使用场景、核心目的、叙事构思、使用方式、内容边界
- **封面页**：建立主题
- **背景变化页**：为什么现在讨论
- **问题/风险页**：让听众看到问题和影响
- **框架/方法页**：结构化理解
- **方案页**：如何解决
- **产品映射页**（可选）：客户问题 → 能力 → 产品 → 价值
- **路线图页**（可选）：分阶段建设
- **总结/行动建议页**：核心结论 + 优先建议 + 下一步
- **Thank You 页**（必有）：简洁收尾，可含联系方式

---

## 页面草图常用样式

左右对比页 / 三段式逻辑页 / 流程页 / 能力地图页 / 路线图页。Wireframe 用 ASCII 文本框表达层级即可。

---

## 通用编写原则

1. 每页只解决一个核心问题
2. 每页有明确角色与承接关系
3. 主线页保持简洁，复杂内容放附录
4. 不要写成文章，不要一页讲所有
5. 重要概念先定义再展开
6. 关键判断要有支撑，不能只堆结论
7. 不服务主线的内容应删除、合并或下沉到附录
8. 每页都有 Speaker Notes，服务口头表达，不重复 bullet
9. 依赖假设/数据/外部事实的内容标明来源或【待确认】

---

## 最终检查清单（交付前必查）

**故事逻辑**
- [ ] 有清晰主线，回答了为什么讲、为什么重要、为什么相信、怎么办？
- [ ] 页面间有因果与递进，不是并列罗列？
- [ ] 自然落到行动？

**内容完整性**
- [ ] 主题、听众、场景、目的、范围边界明确？
- [ ] 已确认 / 待确认 / 不在范围已区分？
- [ ] 关键术语统一？

**页面结构**
- [ ] 第一页隐藏说明页？最后一页 Thank You？
- [ ] 每页有标题、角色、主张、内容、Speaker Notes、Wireframe、注意事项？
- [ ] Speaker Notes 可直接讲述，且说明承接关系？

**后续生成参考**
- [ ] 已写明后续 Agent 的硬约束（不改标题/内容/顺序、保留隐藏页、原生表格、可编辑 pptx、16:9、浅色、微软雅黑）？

**可执行性**
- [ ] 后续 Agent 仅凭此文件即可生成幻灯片，无需追问？
- [ ] 缺失信息已标【待确认】？

---

## 重要提示

- **拿到主题不要直接写规格**——先完成访谈阶段，把背景、叙事、关键内容问清楚。
- 不要直接生成 PPTX——除非用户明确要求（那是后续 Agent 的任务）。
- 不要把规格文件写成普通文章。
- 页数服务故事，不套模板。
- 当任何信息缺失时，宁可标【待确认】也不要编造。
