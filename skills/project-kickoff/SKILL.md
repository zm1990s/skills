# Project Kickoff Skill

访谈式项目初始化。核心方法：**先问清楚，再写契约；先契约，再代码；先模块，再单元。**

**触发词**：新建项目 / 项目初始化 / 从零开始 / 帮我起一个项目骨架 / 想做一个 xxx。

**风格**：中性、专业、直接。不废话，不预设方案，先理解用户处境再收敛。

---

## 一、核心原则（贯穿全流程）

1. **先问后答**：在充分理解 **背景 + 目标 + 约束 + 核心矛盾** 之前，不给方案、不写文件。
2. **每次聚焦一个问题**：`AskUserQuestion` 每轮 questions 数组只放 1 个问题。层层深入，不要一次抛 4 个。
3. **所有澄清必须走 `AskUserQuestion`**：不要在正文里用 markdown 列选项让用户手动选。
4. **适时复述**：每 3~5 轮，用 1~2 句话复述你对用户处境的理解，请对方确认后再继续。
5. **先契约再模块，先模块再单元**：不允许跳过契约直接开始写代码。
6. **每一阶段产出都要经用户确认**才进入下一阶段。

---

## 二、访谈阶段设计

访谈分四个阶段，节奏由浅入深。**不要机械提问**，根据用户上一轮的回答动态选择下一个问题。

### 阶段 A · 意图澄清（1~3 轮）

目标：搞清楚"用户到底想做什么、为什么要做"。

**首轮开场**（用户抛出想法后立即执行）：

- 用 1 句话简短回应（"好，先让我问几个问题理清一下"），**不要**开始设计或列技术栈。
- 立刻用 `AskUserQuestion` 提第一个问题。

**建议的追问维度**（按需选择，不要全问）：

| 维度 | 典型问句 |
|------|---------|
| 项目性质 | 这个项目是给谁用的？ |
| 触发动机 | 为什么现在要做？是市场机会、内部痛点，还是探索性 side project？ |
| 成功标准 | 3 个月后，什么样叫"这个项目做成了"？ |
| 时间/预算约束 | 你有多长时间？是一个人做还是有团队？ |
| 复用 vs 从零 | 是要复用现有系统的一部分，还是完全从零？ |

**Tips**：如果用户已经在开场里说了"我要做一个 Web 应用 + Python 后端"，**不要**再问项目类型和技术栈，直接从"用户/场景"开始问。

### 阶段 B · 范围与形态（2~4 轮）

目标：把 MVP 边界和技术形态定下来。

| 维度 | 典型 options |
|------|-------------|
| 项目类型 | Web 全栈 / 纯后端服务 / CLI 工具 / 数据/AI Pipeline / 桌面或移动端 / 其他 |
| 后端技术栈 | Python+FastAPI / Node.js+Next.js / Go+Gin / 其他 |
| 前端形态 | Next.js+Tailwind / React+Vite / Vue3 / 无前端 |
| 数据存储 | PostgreSQL / PostgreSQL+pgvector / MySQL / SQLite / 无数据库 |
| 部署形态 | Docker Compose / 本地脚本 / K8s / Serverless |
| LLM 集成 | 无 / 有（进一步问：统一网关 or 直连 SDK） |
| 认证需求 | 无 / 单密码门 / JWT+多用户 |

**Tips**：**推荐默认值**要说出来（"通用推荐 Postgres"），但让用户明确否决或接受，不要替他决定。

### 阶段 C · 模块与优先级（2~3 轮）

目标：让用户自己讲出这个项目应该有哪些"模块"，然后一起排优先级。

问句示例：

- "如果把这个系统切成 3~5 个模块，你会怎么切？"
- "这些模块里，哪个是最先必须能跑通的（MVP 核心路径）？"
- "有没有哪个模块你其实还没想清楚、心里没底？"

如果用户答不上来，你可以主动提议模块划分方案，让用户选一个作为起点。

### 阶段 D · 复述与确认（1 轮）

在动手写文件之前，**必须**做一次结构化复述：

```
基于我对你需求的理解：
- 项目目标：______
- 用户与核心场景：______
- 技术栈：______
- MVP 边界：______
- 模块划分：______
- 硬约束（时间/资源/合规）：______

如果这个理解基本正确，我就开始生成契约三件套、PRD 表格和模块化实现路径。
```

用 `AskUserQuestion` 请用户确认：`确认无误 / 需要修正某项 / 大方向对但细节要调整`。

---

## 三、AskUserQuestion 使用规范

- `questions` 数组每次只放 1 个问题。
- `header`：2~4 字标签（如"用户群"、"技术栈"、"核心模块"）。
- `question`：完整问句，以"？"结尾。
- `options`：2~4 个选项，`label` ≤5 字，`description` 1 句话说清含义。
- 最后一个选项通常是"其他/让我详细说说"，鼓励用户在必要时补充。
- `multiSelect`：默认 false；"有哪些模块"、"想引入哪些能力"等场景可设为 true。

## 四、交付物规格（阶段 D 确认后执行）

一次性生成以下 **4 类文件**。每个文件都基于访谈结果精确填充，禁止使用占位符敷衍。

### 交付物 1 · 工程契约三件套

#### 4.1 `CLAUDE.md`

```markdown
# 项目记忆

## 身份
你是 [项目名] 的架构助手。任何改动前先给计划，等用户确认再执行。

## 技术栈（禁止中途更换，如需变更走 /design 流程）
- Backend: [语言+框架+版本]
- Frontend: [框架+版本] / 无
- 数据库: [数据库+版本]
- LLM 接入: [统一网关方案 / 无]
- 部署: [方案]

## 目录约定
[根据项目类型生成，例如：]
backend/
  app/
    api/        # 路由层，只做参数校验与转发
    services/   # 业务逻辑，单一职责
    models/     # ORM 模型
    schemas/    # 请求/响应 schema (Pydantic)
    core/       # 配置、DB 连接、鉴权中间件
    gateway/    # [若有 LLM] 唯一 LLM 出口
  tests/

## 硬性禁止
- 禁止修改 .env（只改 .env.example）
- 禁止直连外部 LLM SDK/API（如有 LLM，只走 gateway/）
- 禁止修改已建好的数据库表结构（只做 additive 增量）
- 禁止把 API key/密钥硬编码进任何代码文件
- 禁止在 api/ 层写业务逻辑
- 禁止把多个功能塞进一个巨大文件

## 强制流程
- 改动前：先给计划，等用户确认再改文件
- 改动后：自己跑 lint + 测试验收
- 新增依赖：先询问是否必须，列出替代方案
- 遇到不确定的地方：**明确说"我不确定"**，不要硬猜
```

#### 4.2 `DESIGN.md`

```markdown
# 架构设计

## 系统架构（文字版）
[组件 A] → [组件 B] → [数据存储]
（附上关键调用链的一句话说明）

## API 契约（一旦确定，后续步骤不得私自修改路径与语义）

| 方法 | 路径 | 说明 | 请求体关键字段 | 返回关键字段 |
|------|------|------|--------------|-------------|
| POST | /[资源] | 创建 | ... | {id, ...} |
| GET  | /[资源]/{id} | 单条查询 | - | {...} |
| GET  | /[资源] | 列表 | ?page&size | [{...}] |

## 数据模型（一旦确定，后续只做 additive 修改）

### [表名]
| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | UUID | PK | 主键 |
| created_at | TIMESTAMPTZ | NOT NULL, default now() | 创建时间 |
| ... | ... | ... | ... |

## 关键技术选型的"为什么"
（记录为什么选了 X 而不是 Y，避免后续被人推翻重来）

## 关键约束
- [性能约束]
- [安全/合规约束]
- [兼容性约束]
```

#### 4.3 `WORKFLOW.md`

```markdown
# 开发阶段协议

## /refine（需求澄清）
输入：需求描述
输出：更新 docs/PRD.md（目标用户、功能边界、验收标准）
通过条件：每条验收标准可观察、可量化

## /design（架构设计）
输入：PRD.md
输出：更新 DESIGN.md（API 契约、数据模型）
通过条件：所有端点有明确入参/出参；所有表字段有类型和约束

## /plan（实现计划）
输入：DESIGN.md + ROADMAP.md
输出：分步实现计划（每步 ≤200 行改动）
通过条件：计划经用户确认

## /build（编写代码）
输入：计划
输出：代码文件 + 测试
通过条件：lint 通过 + 测试绿

## /review（代码审查）
输入：diff
输出：两阶段报告（Spec Compliance → Code Quality）
通过条件：无 Critical 问题

## /ship（发布）
输入：通过 review 的代码
输出：部署配置 + 上线检查
通过条件：健康检查通过
```

### 交付物 2 · PRD 需求表格（`docs/PRD.md`）

**必须**是结构化表格，禁止只写散文段落。

```markdown
# PRD · [项目名]

## 1. 目标用户与核心场景
| 用户角色 | 使用场景 | 期望结果 |
|---------|---------|---------|
| ... | ... | ... |

## 2. MVP 功能需求表（核心）

| ID | 模块 | 功能 | 描述 | 优先级 | 验收标准（可量化） | 依赖 |
|----|-----|------|------|--------|-------------------|------|
| F-01 | 用户 | 注册 | 邮箱+密码注册 | P0 | 提交合法邮箱 → 返回 201 + user_id；重复邮箱返回 409 | - |
| F-02 | 用户 | 登录 | 邮箱+密码登录 | P0 | 正确凭据返回 JWT；错误凭据返回 401 | F-01 |
| ... | ... | ... | ... | ... | ... | ... |

**优先级定义**：
- P0：MVP 必须，砍掉就不能发布
- P1：MVP 强烈建议，砍掉体验会明显打折
- P2：MVP 之后再做，本次不投入

## 3. 明确的 Out of Scope（本次不做）
- ...
- ...

## 4. 待确认问题（诚实标注）
| # | 问题 | 影响 | 何时需要决策 |
|---|------|------|------------|
| Q1 | ... | ... | ... |

## 5. 非功能需求
- 性能：...
- 安全：...
- 可维护性：...
```

### 交付物 3 · 模块化实现路径（`docs/ROADMAP.md`）

**这是本 skill 的差异化核心**。每个模块都拆到可独立交付的实现单元（Implementation Unit），每个 Unit 满足"独立可测、独立可回退、独立可 review"。

```markdown
# 实现路径 · [项目名]

## 模块总览
| 模块 ID | 模块名 | 核心职责 | 优先级 | 依赖模块 |
|---------|-------|---------|--------|---------|
| M1 | 认证 | 用户注册/登录/会话 | P0 | - |
| M2 | ... | ... | P0 | M1 |
| ... | ... | ... | ... | ... |

---

## 模块 M1 · 认证

**目标**：提供邮箱+密码的注册、登录、JWT 会话管理。

**产出契约**：
- API：POST /auth/register, POST /auth/login, GET /auth/me
- 数据表：users, sessions
- 中间件：require_auth

**实现单元清单**：

| Unit ID | 名称 | 内容 | 预估改动 | 验收 |
|---------|------|------|---------|------|
| M1-U1 | users 表 + 迁移 | 建表 SQL，字段严格按 DESIGN.md | ~30 行 | 迁移可跑通，desc users 字段齐全 |
| M1-U2 | user schema + model | Pydantic schema 与 ORM model | ~50 行 | mypy/pyright 通过 |
| M1-U3 | 密码哈希工具 | bcrypt 封装 + 单测 | ~30 行 | pytest 通过（含 3 个边界用例） |
| M1-U4 | 注册端点 + 测试 | POST /auth/register + 3 个测试用例 | ~80 行 | happy + duplicate + invalid email |
| M1-U5 | 登录端点 + JWT | POST /auth/login + JWT 签发 | ~80 行 | 正确凭据返 JWT，错误返 401 |
| M1-U6 | require_auth 中间件 | 校验 JWT，注入 current_user | ~40 行 | 无 token 返 401，过期返 401 |
| M1-U7 | /auth/me 端点 | GET /auth/me，返回当前用户 | ~30 行 | 有 token 返用户，无 token 返 401 |

**M1 完成标准**（Definition of Done）：
- [ ] 所有 Unit 完成且测试绿
- [ ] 集成测试：注册→登录→带 token 访问 /auth/me 全链路通
- [ ] API 路径与 DESIGN.md 100% 一致
- [ ] 无直连外部 SDK（如 LLM）

---

## 模块 M2 · [下一模块]
（同上格式）

---

## 交付顺序建议

**Phase 1（周 1）**：M1 认证 全部 Units
**Phase 2（周 2）**：M2 核心业务 U1~U4
**Phase 3（周 3）**：M2 剩余 + M3 开始
...

每完成一个 Unit，建议 `git tag M1-U3-done`，方便回退。
```

**Unit 设计的黄金法则**：
- 每个 Unit **改动 ≤200 行**（含测试）
- 每个 Unit **单独可 review**（一个 PR/commit）
- 每个 Unit **有明确验收命令**
- 每个 Unit **失败不影响已完成的 Unit**（可回退）
- **绝对不要**把"实现整个认证模块"当作一个 Unit

### 交付物 4 · 分步系统提示词（`prompts/` 目录）

为后续每一步生成可直接复制给 Claude Code 的提示词。文件命名：`prompts/step-N-<name>.md`。

#### `prompts/step-0-contract.md`

```
你正在初始化项目 [项目名]。请只做以下事情，不写任何业务代码：

1. 按照 CLAUDE.md 的目录约定创建空目录结构
2. 生成 .env.example，列出所有需要的环境变量占位（值留空）
3. 生成 .gitignore（含 .env、__pycache__/、node_modules/ 等）
4. 生成 README.md 骨架（含项目简介、快速启动、技术栈）
5. 初始化 git 仓库

不允许：写任何 .py/.ts/.sql 业务文件。
完成后跑：
  ls *.md .env.example .gitignore
  git status --short
输出验收结果。
```

#### `prompts/step-1-scaffold.md`

```
基于 CLAUDE.md 的目录约定和 DESIGN.md 的数据模型，生成项目骨架。

先列完整目录树让我确认，再生成关键文件：
- [若有 Docker] docker-compose.yml：[列出服务]
- backend 分层目录 + Dockerfile + 依赖配置
- [若有前端] frontend 骨架 + Dockerfile
- [若有数据库] infra/[db]/init.sql：严格按 DESIGN.md 建表，不要自创列
- Makefile：dev / down / test

不允许写业务逻辑代码。
完成后跑：
  [docker] docker compose config >/dev/null && echo compose 合法
  find [backend] -maxdepth 3
```

#### `prompts/step-2-M1-auth.md`（按模块生成，一个模块一个文件）

```
实现 ROADMAP.md 中的 模块 M1 · 认证。

**你必须按 Unit 逐个推进**：
- 一次只做一个 Unit
- 每个 Unit 做完先跑测试，绿了再进下一个
- 每个 Unit 完成后停下来向我汇报，我确认后你才能继续

按顺序：M1-U1 → M1-U2 → M1-U3 → M1-U4 → M1-U5 → M1-U6 → M1-U7

严格约束：
- API 路径必须与 DESIGN.md 完全一致，不得改名
- schema/model 是唯一数据来源，不允许在别处散落 dict
- 每个端点必须配 pytest（至少 1 个 happy + 1 个 error path）

现在开始 M1-U1，先给我实现计划，我确认后再改文件。
```

#### `prompts/step-N-review.md`（治理层）

```
1. 创建 .claude/hooks/pre-tool-use-guard.sh：拦截 rm -rf、修改 .env、DROP TABLE 等危险命令
2. 创建 .claude/hooks/post-tool-use-lint.sh：对改动过的源码文件跑 lint
3. 创建 .claude/agents/reviewer.md：两阶段 Review（Spec Compliance → Code Quality）
4. 创建 .claude/settings.local.json 注册 hooks
5. 用 reviewer 对当前代码跑一次 review，输出报告
```

## 五、生成阶段的执行动作

阶段 D 用户确认后，按如下顺序**在一次交互内**完成：

1. 在**同一个 assistant 回复**里，一次性生成上述 4 类文件到 `docs/`、`prompts/` 目录及项目根。
2. 生成完成后，用 present_files 呈现给用户。
3. 附一段"下一步指引"：
   - 推荐用户先 `git init && git add . && git commit -m "chore: kickoff contract & roadmap"` 打上基线
   - 然后按 `prompts/step-0-contract.md` 开始，逐步执行
   - 提醒用户：每完成一个 Unit 建议 `git tag M<N>-U<N>-done`

## 六、常见反模式（明确禁止）

1. ❌ 用户一开口就直接抛出技术栈方案 → **必须先访谈**
2. ❌ 一次问 4 个问题 → 每轮只问 1 个
3. ❌ PRD 只写散文没表格 → **必须结构化**
4. ❌ 模块只列名字没拆 Unit → 每个模块必须拆到可实现单元
5. ❌ 一个 Unit 写 500 行 → 单元大小失控就重新拆
6. ❌ 直接开始写代码 → 契约三件套 + PRD + ROADMAP + prompts 是前置
7. ❌ 用占位符敷衍（"TODO"、"待补充"）→ 访谈信息不足时应回到访谈，而非留 TODO

[AskUserQuestion header="是否确认" question="这个理解基本正确吗？"

## 八、成功标准（本 skill 自身）

用户走完一次 `/project-kickoff` 后，应当拿到：

1. ✅ 一份自己认可、边界清晰的 PRD 表格
2. ✅ 一套约束后续代码改动的工程契约
3. ✅ 一份可以照着一个 Unit 一个 Unit 推进的实现路径
4. ✅ 一叠可直接复制给 Claude Code 的分步提示词
5. ✅ 心里对"3 个月后做成什么样"有明确画面
